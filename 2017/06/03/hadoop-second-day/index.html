如何上传文件？

过程：a.log(200M),client去请求NameNode，我要上传文件;NameNode维护着一些元数据(记录机器存贮容量等信息，账本)，NameNode计算第二个DataNode可以存贮a.log，于是NameNode把信息返给client，client将a.log写入第二个DataNode。

如何保证数据安全？

冗余存贮。DataNode与DataNode之间进行数据的复制，直到达到配制的副本数量。当写入dataNode的一个block完成，就会进行水平复制，复制完成还会反馈回去，NameNode进行记录。

元数据的存贮位置

选择存贮在NameNode节点而不是放在client端，client端会随时关闭，导致元数据丢失，找不到数据位置

数据的分块

hadoop1.x的数据块64M，2.x的是128M。上传是先分块，写完一块再分一块。比如上传1G的文件，分成8块，第一块写完，再写第二块，……不能8块同时写。好处是：我读写的时候，假如读到第八块失败，不用从头再读，只要读第八块就ok了。

什么是分布式文件系统

  • 文件数量越来越多，一台操作系统管辖不了，那就分配到更多的操作系统，但是不方便管理，迫切需要一种系统来管理多台机器上的文件

  • 允许通过网络共享文件存贮空间

  • 用户不用关心底层具体实现
  • 容错。即使某些节点脱机，整个系统仍然可以持续运行
  • 分布式文件系统有很多种，hdfs只是其中一种，智用于一次写入多次读取，不支持并发写入，小文件不适合。
    • 并发写：数据块不能同时写的意思
    • 小文件：小文件也要占据一个元数据(内存中)，导致瓶颈

HDFS的shell操作

  1. 查看帮助
    hadoop fs -help <cmd>
  2. 上传
    hadoop fs -put <linux上文件> <hdfs上的路径>
  3. 查看文件内容
    hadoop fs -cat <hdfs上的路径>
  4. 查看文件列表
    hadoop fs -ls /
  5. 下载文件
    hadoop fs -get <hdfs上的路径> <linux上文件>

HDFS的架构

HDFS架构由三个角色组成，分别是NameNode,DataNode,Secondary NameNode;各自的作用分别是

  1. NameNode。管理者，管理DataNode干活
  2. DataNode。存贮数据
  3. Secondary NameNode。可以理解为NameNode的助理，帮助NameNode干活。它并不是NameNode的热备，不能代替NameNode干活，NameNode宕机，也不会自动切换到它上面工作，并且NameNode宕机后它也不工作了。
